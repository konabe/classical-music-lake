# ユーザー認証機能 - 要件定義

## 概要

Classical Music Lake にユーザー認証機能を追加し、ユーザーが登録・ログイン・ログアウトできる基盤を構築する。

## 目的

- 複数ユーザーが個別の視聴ログを管理できるようにする
- 認証を通じて個人情報保護を実現する
- 今後のユーザー管理・プロフィール機能の基盤となる

## 機能スコープ

### 実装する機能

1. **ユーザー登録**
   - メールアドレス（ユニーク）
   - パスワード（一定の強度要件）
   - 登録後の確認メール送信

2. **ログイン**
   - メールアドレス + パスワード による認証
   - JWT トークン（アクセストークン）の発行
   - セッション管理

3. **ログアウト**
   - クライアント側での認証情報削除（Cookie と localStorage の両方をクリア）
   - サーバー側トークン失効: Cognito の `GlobalSignOut`（または `AdminUserGlobalSignOut`）を呼び出し、全セッション・リフレッシュトークンを無効化する
   - サーバー側失効が失敗した場合のエラーハンドリングを実装し、ユーザーへの適切なフィードバックを提供する
   - ログアウト操作は監査ログに記録する（いつ・誰が・どのデバイスでログアウトしたか）

### 実装しない機能（将来の実装対象）

- パスワードリセット / 忘れた場合の復旧
- プロフィール編集
- 多要素認証（MFA）
- ソーシャルログイン（OAuth）

## 技術要件

### バックエンド

- AWS Cognito User Pool を使用
- Cognito User Pool の作成・設定
- ユーザー登録・ログイン API との連携

### フロントエンド

- Nuxt 4 でカスタム登録・ログインフォーム を実装
- アクセストークンの保存・管理（Secure Cookie: HTTPOnly、Secure、SameSite=Strict）
  - refresh token は HttpOnly Secure Cookie に保存し、access token は短寿命（60 分以下）で管理する
  - トークンローテーション（refresh token の使用後に新しい refresh token を発行）を必須とする
- API 呼び出し時のトークン付加（Authorization ヘッダー）
- 認証状態に応じた ページルーティング制御

### 既存機能との連携

- 現在の視聴ログ API を認証保護（Authorization ヘッダーを検証）
- 認証されたユーザーのみが視聴ログを取得・作成・編集できるようにする

## 前提条件

- AWS Cognito User Pool が利用可能
- フロントエンド・バックエンド通信は HTTPS
- CORS 設定が適切に行われていること

## 依存関係

- なし（新規追加機能）
