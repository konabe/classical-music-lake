# 管理者ロールの追加 - 要件定義

## 概要

楽曲マスタの書き込み操作（作成・更新・削除）を「管理者ロール」を持つユーザーのみが実行できるようにする。現状は認証有無に関わらず誰でも楽曲マスタを編集できてしまうため、権限分離を導入してマスタデータを保護する。

関連 Issue: konabe/classical-music-lake#476

## 目的

- 楽曲マスタというサービス共通のデータに対する意図しない書き換え・削除を防止する
- 一般ユーザーと管理者を明確に区別し、今後の管理者機能拡張の土台を作る
- 管理者であることを UI 上でも区別し、一般ユーザーには管理系の導線を見せない

## 機能スコープ

### 実装する機能

- 管理者ロールの識別手段を導入し、現在ログイン中のユーザーが管理者かどうかを判定できるようにする
- 楽曲マスタの書き込み API（`POST /pieces`・`PUT /pieces/{id}`・`DELETE /pieces/{id}`）を管理者のみ実行可能にする
  - 非管理者が書き込み API を呼び出した場合は `403 Forbidden` を返す
  - 楽曲マスタの参照 API（`GET /pieces`・`GET /pieces/{id}`）は従来どおり公開（認証・ロール不問）
- フロントエンドで管理者ロール判定に応じた表示切り替えを行う
  - TOP ページの「管理者向けリンクセクション」は管理者ロール保有者のみ表示する
  - 楽曲マスタの新規作成・編集・削除ボタン等の操作 UI は管理者ロール保有者のみ表示する
  - 非管理者が管理者専用ページ（楽曲マスタの作成・編集ページ等）へ直接 URL でアクセスした場合は管理者向け UI を描画せず、適切にリダイレクトまたはアクセス不可表示を行う
- 管理者の付与・剥奪は AWS コンソール／CLI での手動運用とする（画面からの管理機能は MVP では実装しない）

### 実装しない機能（将来の実装対象）

- 管理者による他ユーザーの閲覧／BAN／ロール付与などの GUI 機能
- 視聴ログ・コンサート記録など、ユーザー個人データに対する管理者モデレーション
- 楽曲マスタ参照 API の認証化・ロール制限
- 複数ロールや階層的な権限（例: 編集者・閲覧者・スーパー管理者）
- 管理者操作の監査ログ

## 設計申し送りメモ

- ロール管理方式は AWS Cognito User Pool の Groups 機能を採用する方針（`admin` グループを新設し、`cognito:groups` クレームで判定）
- 運用手順: AWS コンソールまたは CLI（`aws cognito-idp admin-add-user-to-group`）で対象ユーザーを `admin` グループに追加する。手順は `docs/OPERATIONS.md` へ追記する
- 認可の強制箇所は API Gateway / Lambda のどちらで行うか設計時に決める（ID Token / Access Token のクレーム位置差異も考慮）
- フロントエンドでは既存の `useAuth` コンポーザブルに「管理者かどうか」を返す API を追加する前提で設計する
- 非管理者が管理者向け操作を試みた場合:
  - API 側: `403 Forbidden` をエラーレスポンスとして返す
  - フロント側: 管理者向けリンクセクションおよび楽曲マスタ書き込み導線を非表示にする（存在しないように見せる）
- 既存ユーザーは全員「一般ユーザー」扱いとなる。初期の管理者アカウントは運用者が手動で `admin` グループへ追加する想定
- `SPEC.md` / `ARCHITECTURE.md` に管理者ロールおよび 403 レスポンス仕様、TOP ページの表示差分を反映する
