# Work 014-01: 楽曲マスタ一覧のページネーション対応 - 要件定義

関連 Issue: konabe/classical-music-lake#295
親要件: [`../requirement.md`](../requirement.md)

## 概要

楽曲マスタ（Piece）の一覧取得 API・フロントエンド画面にカーソル型ページネーションと無限スクロール UI を導入する。`GET /pieces` のレスポンス形式を変更し、`/pieces` 一覧画面をスクロールに応じて次ページを継ぎ足し表示する。

## ユーザーストーリー

> 楽曲マスタの利用者として、登録件数が増えても一覧画面の初期表示が重くならず、スクロールしながら続きを自然に閲覧できるようにしたい。

## 要件

### 振る舞い

- `GET /pieces` はデフォルトで先頭から最大 50 件を返す。
- クライアントがレスポンスの `nextCursor` を次回リクエストの `cursor` に指定すると、続きのページを取得できる。前ページとの重複は発生しない。
- 全件取り切った時点で `nextCursor` は `null`（または未設定）となる。
- `/pieces` 一覧画面では、初期表示で 1 ページ目を取得し、画面末尾までスクロールすると次ページを自動で取得・追記する。
- 末尾までスクロールしたが `nextCursor` が無い場合は以降追加リクエストを行わない。
- ページ読み込み中はローディング表示、失敗時はエラーメッセージと再試行手段を表示する。
- ソート順は現行挙動（DynamoDB Scan の戻り順・順不同）を維持する。
- 楽曲詳細画面・コンサート記録作成画面など楽曲一覧を「全件前提」で利用している画面は、従来通り全件を扱えること。

### 入出力

- `GET /pieces` クエリパラメータ
  - `limit`（任意）: 1 ページあたりの件数。既定 50、上限 100。範囲外は `400 Bad Request`。
  - `cursor`（任意）: 続きを取得する際のカーソル文字列。不正値は `400 Bad Request`。
- `GET /pieces` レスポンス

  ```json
  {
    "items": [
      /* Piece の配列（最大 limit 件） */
    ],
    "nextCursor": "..." // 次ページが無い場合は null
  }
  ```

  - カーソル文字列の中身はクライアント非公開（不透明）。

### 影響範囲（機能観点）

- `/pieces` 一覧画面のスクロール挙動が変わる（無限スクロール）。
- `GET /pieces` のレスポンスが配列から `{ items, nextCursor }` に変わる（破壊的変更）。
- 楽曲を全件前提で扱う画面（楽曲詳細、コンサート記録作成の楽曲選択 UI 等）の挙動は従来通り。

## スコープ外（このワークで扱わないこと）

- 視聴ログ・コンサート記録のページネーション対応
- 新規 GSI の追加
- 楽曲マスタの検索・フィルタリング
- ページ番号 UI、「もっと読み込む」ボタン UI
- URL への `cursor` の反映（ブラウザバック時のページ状態復元）

## 受け入れテストケース

### API 受け入れ

- [ ] `GET /pieces` を `limit` 無指定で呼ぶと、最大 50 件の `items` と必要に応じた `nextCursor` が返る。
- [ ] `GET /pieces?limit=10` で呼ぶと、最大 10 件の `items` が返る。
- [ ] `limit` に 0 や負数、101 以上の値を指定すると `400 Bad Request` が返る。
- [ ] レスポンスの `nextCursor` を次回リクエストの `cursor` に指定すると、続きのページが返り、前ページの `items` と重複しない。
- [ ] 全件取得しきった最後のレスポンスでは `nextCursor` が `null`（または未設定）となる。
- [ ] `cursor` に不正文字列を渡すと `400 Bad Request` が返る。
- [ ] 登録件数が 0 件の場合、`items: []`, `nextCursor: null` が返る。
- [ ] ページング有無にかかわらず、返却される Piece オブジェクトのフィールドは従来仕様と同じ（`id`, `title`, `composer`, `videoUrl?`, `genre?`, `era?`, `formation?`, `region?`, `createdAt`, `updatedAt`）。

### フロントエンド受け入れ

- [ ] `/pieces` を開くと 1 ページ目（最大 50 件）が表示される。
- [ ] 画面を下方向にスクロールし一覧末尾に達すると、追加ページが自動取得され一覧に追記される。
- [ ] 最終ページまで到達した後はスクロールしても追加リクエストが発生しない。
- [ ] 読み込み中はローディングインジケーターが表示される。
- [ ] 読み込みに失敗すると、エラーメッセージと再試行操作（ボタン等）が表示され、再試行で続きを取得できる。
- [ ] 楽曲詳細画面（`/pieces/[id]`）、コンサート記録新規作成画面（`/concert-logs/new`）など、楽曲一覧を全件前提で利用している画面が従来通り動作する。
- [ ] 既存の表示コンポーネント（`PieceList` など）の見た目は変わらない（ページ切り替えの違和感がない）。

### ドキュメント

- [ ] `docs/SPEC.md` の楽曲マスタ API セクションにページング仕様（`limit`・`cursor`・`{ items, nextCursor }`）が記載されている。
- [ ] `docs/SPEC.md` の「制限事項」にあった「ページネーションなし（フロント向け）」の記述が、楽曲マスタについては解消済みであることを反映している。
