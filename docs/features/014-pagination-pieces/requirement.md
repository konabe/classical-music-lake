# 楽曲マスタのページネーション - 要件定義

関連 Issue: konabe/classical-music-lake#295

## 概要

楽曲マスタ（Piece）の一覧取得 API・フロントエンド画面にページネーションを導入する。バックエンドは DynamoDB の Scan を継続しつつカーソル型ページングを返却し、フロントエンドは「無限スクロール」でページを継ぎ足し表示する。

## 目的

- データ量増加時の一覧取得のレスポンスタイム・転送量を抑え、UX とコストを安定させる。
- 将来的な他一覧 API（視聴ログ・コンサート記録）にも転用可能な共通ページング仕様を先行して確立する。

## 機能スコープ

### 実装する機能

- **楽曲マスタ一覧 API (`GET /pieces`) のカーソル型ページネーション**
  - クエリパラメータ
    - `limit`: 1 ページあたりの件数。省略時は 50 件。最大 100 件を上限とする。
    - `cursor`: 前回レスポンスで返却された `nextCursor` を指定して続きを取得する。省略時は先頭から取得。
  - レスポンス形式（既存の配列直返しから変更する点に注意）
    ```json
    {
      "items": [
        /* Piece の配列（最大 limit 件） */
      ],
      "nextCursor": "..." // 次ページが無い場合は null または未設定
    }
    ```
  - カーソルの中身は DynamoDB の `LastEvaluatedKey` を Base64 エンコードした不透明文字列とし、クライアントはその中身に依存しない。
  - ソート順は現行の Scan の戻り順（順不同）を維持する。
- **フロントエンド楽曲マスタ一覧画面 (`/pieces`) の無限スクロール**
  - 画面初期表示時に 1 ページ目を取得する。
  - 一覧末尾にスクロールが達したら次ページを自動取得し、末尾に追記する。
  - 末尾到達の検出は IntersectionObserver などブラウザ標準機能で行う。
  - `nextCursor` が無くなった時点でそれ以降の読み込みを停止する。
  - 読み込み中はローディング表示を行う。
  - 取得失敗時はエラーメッセージを表示し、リトライ手段（ボタン等）を提供する。
- **`usePieces` composable の更新**
  - カーソル型の取得に対応したメソッドを提供する。
  - 既存の一覧画面以外で `usePieces` を利用している箇所（`/pieces/[id]/index.vue`、`/concert-logs/new.vue` などで楽曲選択に利用）は、一覧全件取得が前提の振る舞いを維持できるよう、無限スクロール用 API と全件取得用ヘルパーを両立させる。

### 実装しない機能（将来の実装対象）

- 視聴ログ (`GET /listening-logs`)・コンサート記録 (`GET /concert-logs`) のページネーション対応。今回の API 仕様は将来これらにも転用可能な形とするが、実装は本 Work では行わない。
- 新規 GSI の追加（`userId + listenedAt` や `composer` などのソートキー用インデックス）。
- タイトル・作曲家などでの検索・フィルタリング機能。
- ページ番号型 UI（ページ 1 / 2 / 3...）や「もっと読み込む」ボタン型 UI。今回は無限スクロールのみ採用する。
- SSR / SEO 向けの URL にページ状態を保持する仕組み（`?cursor=...` をブラウザ URL に反映する等）。

## 受け入れテストケース

### API

1. `GET /pieces` を `limit` 無指定で呼ぶと、最大 50 件の `items` と必要に応じた `nextCursor` が返る。
2. `GET /pieces?limit=10` で呼ぶと、最大 10 件の `items` が返る。
3. `limit` が上限（100）を超える値を指定した場合は `400 Bad Request` となる（または上限値にクランプされる。API 設計で確定する）。
4. 返ってきた `nextCursor` を次回リクエストの `cursor` に指定して呼ぶと、続きのページが返り、前ページとの重複が発生しない。
5. 最後のページでは `nextCursor` が `null`（または未設定）となる。
6. 不正な `cursor` 文字列を指定すると `400 Bad Request` が返る。
7. 登録件数が 0 件の場合は `items: []`, `nextCursor: null` が返る。

### フロントエンド

1. `/pieces` を開くと 1 ページ目（最大 50 件）が表示される。
2. 画面をスクロールして一覧末尾に到達すると、自動的に次ページが読み込まれ一覧に追記される。
3. `nextCursor` が無くなった以降はスクロールしても追加リクエストが発生しない。
4. 読み込み中はローディングインジケーターが表示される。
5. ネットワークエラー時はエラーメッセージが表示され、再試行操作が可能である。
6. 楽曲詳細・コンサート記録作成など、楽曲一覧を全件前提で利用している画面が従来通り動作する。

## 設計申し送りメモ

- レスポンス形式が配列から `{ items, nextCursor }` へ変わるため、互換性を持たせず API バージョンは据え置きで破壊的変更として扱う。呼び出し側（`usePieces` 等）をまとめて更新する。
- カーソルは DynamoDB `LastEvaluatedKey` の Base64 エンコードとする。デコード失敗時のエラーハンドリング（`InvalidCursor`）を実装すること。
- `limit` のデフォルト 50・上限 100 は API・フロントエンド双方で定数として共有できる形にする（`shared/constants.ts` への定数追加が自然）。
- 視聴ログ・コンサート記録への将来展開を考慮し、バックエンドではページング用の共通ユーティリティ（例: `paginateScan`, `encodeCursor`, `decodeCursor`）をユースケース層またはリポジトリ層の再利用可能な形で切り出す。
- 無限スクロールの末尾検出は SSR 無効（SPA）環境で動作すれば十分なため、IntersectionObserver を前提とする。
- `usePieces` の現行利用箇所（楽曲選択 UI など全件前提の画面）との両立方針は、Work 分割時に議論する（全件取得ヘルパーを別途用意する／呼び出し側で nextCursor を辿る等）。
- 変更は `docs/SPEC.md`（4.x 楽曲マスタ API と 9.1 制限事項）に反映する。
