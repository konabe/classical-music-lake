# コンサート記録機能 - 要件定義

## 概要

クラシック音楽のコンサート（演奏会）に行った体験を記録・管理する機能を新規追加する。既存の視聴ログ（CD・配信の記録）とは完全に独立したエンティティとして管理する。

## 目的

- コンサートに行った記録を体系的に蓄積し、後から振り返ることができるようにする
- プログラム（演奏曲目）を楽曲マスタと連携させ、どの曲をコンサートで聴いたかを構造化して記録する
- 視聴ログ（CD・配信）とは別軸で、ライブ演奏の鑑賞体験を管理する

## 機能スコープ

### 実装する機能

- **コンサート記録の CRUD**（作成・一覧・詳細・編集・削除）
- **記録項目**:
  - 日時（コンサートの開催日時）
  - 会場（テキスト自由入力）
  - プログラム（楽曲マスタから複数選択）
  - 指揮者（テキスト自由入力）
  - オーケストラ / アンサンブル（テキスト自由入力）
  - ソリスト（テキスト自由入力）
- **認証**: ログインユーザーのみがアクセス可能（視聴ログと同じ認証方式）
- **ユーザー別データ管理**: ログインユーザーごとにデータを分離
- **DynamoDB テーブル**: 視聴ログとは別テーブルで管理

### 実装しない機能（将来の実装対象）

- 評価（5段階）・お気に入り・メモの記録
- コンサート記録の検索・フィルタリング
- 座席情報・チケット価格の記録
- コンサートホールのマスタ管理
- コンサート記録と視聴ログの横断的な統計・分析

## 設計申し送りメモ

- 既存の視聴ログ（ListeningLog）の設計パターン（DynamoDB テーブル構成、Lambda 関数構成、API Gateway ルーティング、Cognito Authorizer）を踏襲すること
- プログラム（演奏曲目）は楽曲マスタ（Piece）の ID を配列で保持する設計を想定（詳細は設計フェーズで決定）
- 指揮者・オーケストラ・ソリストはそれぞれ独立したテキストフィールドとする
- フロントエンドのページ構成・ナビゲーションは既存パターン（視聴ログ・楽曲マスタ）に合わせること
