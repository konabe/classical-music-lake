# コードレビューガイドライン

## レビューの目的

- バグ・ロジックの誤りを早期に発見する
- コーディング規約・設計方針（DDD）の一貫性を保つ
- 知識共有とコード品質の継続的な向上

---

## レビュアーへ

### 確認すべき観点

#### 正確性・安全性

- [ ] ロジックにバグや考慮漏れがないか
- [ ] エラーハンドリングが適切か（Lambda は `try/catch` + `internalError()`）
- [ ] `any` 型を使っていないか
- [ ] SQLインジェクション・XSS等のセキュリティリスクがないか

#### 設計・コード品質

- [ ] DDD の設計思想に沿っているか
- [ ] 責務が適切に分離されているか（1ファイル1エンドポイント等）
- [ ] 過剰な複雑さや不要な抽象化がないか
- [ ] 型定義が適切か（`Rating` 等の制約型を活用しているか）

#### テスト

- [ ] 新規ロジックにユニットテストが追加されているか
- [ ] バリデーション・異常系のテストが含まれているか

#### ドキュメント

- [ ] 実装を含む変更で `docs/SPEC.md` が更新されているか
- [ ] 複雑なロジックにコメントがあるか

---

## レビュイーへ

### PRを出す前に

- `pnpm test`（フロントエンド）と `pnpm -C backend test`（バックエンド）を実行してテストが通ることを確認する
- ESLint・Prettier のエラーがないことを確認する（pre-commit フックで自動チェック）
- PRテンプレートのチェックリストをすべて確認する

### レビューコメントへの対応

- 指摘には `resolve` ではなく実際に修正してから `resolve` する
- 疑問点や反論がある場合はコメントで議論する

---

## レビューの優先度

| ラベル       | 意味                     |
| ------------ | ------------------------ |
| `blocking`   | マージ前に必ず対応が必要 |
| `suggestion` | 対応推奨だが任意         |
| `nit`        | 細かい指摘（スタイル等） |
